Data Abstraction

• Definition: Data abstraction is the process of hiding irrelevant or complex implementation details from users while providing only the essential information needed to interact with the system.
• Primary Goal: The main objective of a database is to allow for the convenient and effective storage and retrieval of data.
• Analogy for Understanding: Consider a delay in milk delivery; a customer wants a "short and sweet" explanation (abstract information), not a complex, minute-by-minute timeline of the delivery person's morning routine.
• Significance: Users should not need to understand complex underlying data structures to use the system,. For example, Gmail users do not need to know where their data is physically stored or what data models are being used.

The Three Levels of Abstraction

The system is divided into three levels to separate the user’s experience from the physical storage of data.
1. Physical Level (Lowest Level)
• Focus: This level describes how the data is actually stored in the physical medium.
• Storage Media: It involves physical hardware like hard disks or SSDs.
• Implementation Details: It deals with low-level, complex data structures and decides the specific location in memory where data resides.
• Management: This level determines if the database will be centralized or distributed and is primarily the responsibility of Database Administrators (DBAs).
2. Logical or Conceptual Level (Middle Level)
• Focus: This level describes what data is stored in the database and what relationships exist among that data.
• Structure: It defines the entire structure of the database, such as creating tables for students, faculty, or books, and defining their attributes (e.g., account numbers, names, or roll numbers).
• Data Structures: While the physical level uses complex structures, the logical level typically uses simpler data structures.
• Management: This level is handled by DBAs, programmers, and developers who write the application logic.
3. View Level (Highest Level)
• Focus: This level describes how the user interacts with the system and provides different "views" of the data for different users.
• User Interaction: Users interact with the system through a Graphical User Interface (GUI), such as a web or mobile interface.
• Hiding Complexity: This level hides the most complexity; for instance, a user sees a button to "transfer funds" but does not see the background logic or the physical storage processes.
• Multiple Views: Different users see different subsets of the same central database based on their roles.

Key Concepts and Benefits

• Physical Data Independence: This occurs when changes to the logical structure of the database do not require a rewrite of the physical storage structure,. The application programs remain independent of the physical schema.
• Security and Access Control: Security is enforced at the view level by granting specific privileges to different users.
    ◦ Example 1 (University): A student can log in to view their own details, while a faculty member has a different view that includes student attendance and salary information.
    ◦ Example 2 (Banking): A normal user can only see their own account details, whereas a teller, a bank manager, or a regional manager will each have progressively higher levels of access to the same database.
    ◦ Example 3 (ATM): When an ATM card is inserted, the system fetches only the specific details related to that cardholder, providing a unique view for that individual.
