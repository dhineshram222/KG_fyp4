I. Overview of Data Models

• Definition: A data model is the underlying structure of a database and serves as a conceptual tool to describe data, data relationships, data semantics, and constraints.
• Design Levels: Database design occurs at three levels: the physical level, the logical level, and the view level.

II. Five Major Components of a DBMS

A DBMS is comprised of five interrelated components that bridge the gap between machinery and humans:
1. Hardware: Physical electronic devices such as computers and I/O devices.
2. Software: Includes the operating system, DBMS software (e.g., MySQL, Oracle), application programs, and support utilities.
3. Data: The most vital component; it acts as the bridge between the machinery (hardware/software) and the human users.
4. Procedures: The instructions, rules, and logs that govern how the database is used.
5. Users (People): Includes Database Administrators (DBA), designers, software developers, and end-users/customers.

III. Categories of Data Models

1. Relational Model
• Structure: Data is organized into tables (relations) consisting of rows and columns.
• Characteristics: It is a "record-based" model using fixed-format records with unique names for each column and table.
• Status: This is the most widely used model in contemporary database systems.
2. Entity-Relationship (ER) Model
• Structure: A collection of basic objects called entities and the relationships between them.
• Usage: Primarily used for database design as a schematic or diagrammatic representation before the actual database is created.
• Components: Uses entities (real-world objects), attributes (descriptions of the entity), and relationships (how entities interact).
3. Object-Based and Object-Relational Models
• Object-Based: Extends the ER model by adding Object-Oriented Programming (OOP) features like inheritance and data encapsulation. It is ideal for storing multimedia data like audio, video, and images.
• Object-Relational: A hybrid model that combines object-based features with the relational model.
4. Semi-Structured Data Model (XML)
• Structure: Data does not have a fixed schema; individual items of the same type may have different attributes.
• Mechanism: Uses Extensible Markup Language (XML) with user-defined tags to create a hierarchy of parent and sub-elements.
• Usage: Facilitates flexible data transfer between different applications.
5. Obsolete Models
• Hierarchical Model: Organizes data in a tree-like structure with root and child nodes.
• Network Model: Organizes data in a graph-based structure.
• Status: Both are largely obsolete because they are too closely tied to the underlying physical implementation, making data modeling complicated.
